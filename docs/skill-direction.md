# 스킬 방향

## 목적

- 선택된 문서, diff, 문서 패키지를 하나의 분명한 현재 최종 상태로 만든다.
- 문장 수준에서는 표현, 밀도, 요구 언어를 맡고, 패키지 수준에서는 역할, 독자 경로, 현재 소유권을 맡는다.
- 문서 밖의 사실, 정책, 결정, 구현 의미는 만들지 않는다.

## 핵심 방향

- 최신 사용자 지적을 최우선 편집 입력으로 둔다.
- 사용자·저장소 지침, 문서 형식의 전용 계약, 확인된 기준 근거, 일반 문서 원칙 순으로 적용한다.
- 문장 압축이나 패키지 정리 전에 대상 문서의 단일 역할, 핵심 주체, 독자 행동, 책임지는 약속을 먼저 잡는다.
- 이미 구체적으로 요청된 문서 표면을 작성할 때도 사후 교정 메모가 아니라 처음 쓰는 문장에 이 방향을 적용한다.
- 삭제 요청은 실제 삭제로 반영한다.
- 레거시, 예외, 경고, 라우팅은 실제 필요가 확인될 때만 남긴다.
- agent 작업 과정, 판단 보존, 미래 편의, 구현 인접 설명은 대상 문서가 그 역할을 맡을 때만 남긴다.
- 공개 진입점은 확인된 사용자 상황, 결과와 다음 행동을 내부 절차보다 먼저 제시하되 근거 없는 편익을 만들지 않는다.
- 문서의 요구 언어와 로캘을 유지하고, 유지할 비요구 언어 일반 문장은 요구 언어 문장으로 고치며 중복되거나 오래됐거나 여분인 문장은 삭제한다.
- 언어 관련 실패 사례가 특정 언어에서 나왔더라도, 배포 스킬 계약은 “요구된 본문 언어를 지킨다”는 일반 규칙으로 적는다.
- 코드 식별자, API 이름, 경로, 명령어, 제품명은 리터럴로 보존한다.
- 현재 정본을 증거, 시간 순서, 초안, 보관 자료와 분리한다.
- 현재 결정, 계약, 게이트, 계획 하나에는 소유 문서 하나만 둔다.
- 문서의 부분이 독립적인 독자 행동, 소유권, 변경 이유를 가질 때만 분리한다. 독자가 함께 사용하거나 하나의 계약·절차를 완성하는 내용은 함께 두며, 길이·제목 수·개념 이름만으로 쪼개지 않는다.
- 실제 대상 변경에 따른 참조와 동시 수정은 허용한다. 목표는 문서 간 연결을 없애는 것이 아니라 경쟁하는 정본 정의·결정·계약·절차와 불필요한 의미 전파를 줄이는 것이다.
- 패키지 역할은 고정 템플릿이 아니라 진입점, 라우터, 현재 소유 문서, 게이트/런북, 백로그, 증거/로그, 작업/보관 문서를 구분하는 판단 기준으로 쓴다.
- 독자가 재개, 결정, 구현, 검증, 운영 중 무엇을 하려는지에서 문서 경로를 잡는다.
- 라우터는 독자, 선택 기준, 순서, 전제조건이라는 조합 의미를 소유할 수 있지만, 다른 문서의 정본을 복제해 경쟁하지 않는다.
- 소유권이 이동하면 이전 문서는 새 소유 문서로 연결하거나 역사/보관 상태를 밝혀 정본 경쟁을 끝낸다.
- 런북은 사전 조건, 허용 동작, 중단 조건, 승인 경계, 증거, 롤백, 복구가 실제로 사용 가능한지 본다.
- 구조 개선 효과가 작으면 변경하지 않거나 가볍게만 고친다.
- append-only 로그와 유용한 원문 형식은 정리 편의를 위해 다시 쓰지 않는다.
- 기준 소스 선택, 제품·정책·아키텍처 결정, 작업 상태 저장 구조, 구현 계획 의미 변경은 해당 워크플로우로 넘긴다.

## 맡는 질문

- 이 문서가 최신 지적을 실제 본문에 반영했는가.
- 이 문서가 작성 시점부터 요구 언어, 목적, 밀도를 지켰는가.
- 이 문서가 하나의 역할과 독자 행동을 분명히 돕는가.
- 이 경계가 독립적인 독자 행동, 소유권, 변경 이유에 근거하며, 함께 써야 하는 계약이나 절차를 과분할하지 않았는가.
- 삭제 요청이 실제 삭제로 반영되었는가.
- 설명이 직접적이고 필요한 만큼만 남았는가.
- 문서의 요구 언어와 실제 식별자를 구분했는가.
- 이 문서 패키지에서 현재 정본, 증거, 로그, 초안, 보관 자료가 구분되는가.
- 같은 현재 약속을 소유한다고 주장하는 문서가 둘 이상인가.
- 독자가 재개, 결정, 구현, 검증, 운영을 위해 어디서 시작할지 알 수 있는가.
- 소유권 이동 뒤 예전 문서가 여전히 정본처럼 보이는가.
- 라우터가 독자 경로를 설명하되 다른 정본 문서의 내용을 경쟁적으로 복제하지 않는가.
- 게이트나 런북에 중단, 승인, 증거, 롤백, 복구 경계가 빠졌는가.

## 이관 대상

- 코드, API, 정책 중 무엇이 기준 소스인지 판단하는 일.
- 원문에 없는 제품, 정책, 아키텍처, 구현 결정을 만드는 일.
- 작업 상태, 작업 로그, 인수인계 저장 구조를 설계하는 일.
- 구현 계획의 의미나 코드 변경 범위를 바꾸는 일.
- 항상 읽히는 저장소 지침 파일의 의미나 구조를 설계하는 일.
- 순수 번역이나 지역화.

## 수정 기준

- 반복해서 관찰된 편집·패키지 실패만 배포 스킬에 추가한다.
- 스킬 본문은 짧은 실행 계약으로 유지한다.
- 특정 언어권 전용 규칙보다 “요구 언어와 로캘 보존”이라는 일반 규칙을 우선한다. 예시 언어는 관찰 근거이고 적용 범위는 요구 언어 전체다.
- 한국어 페어 문서는 영문 기본 스킬의 의미를 자연스럽게 옮기고 짧은 실행 계약을 유지한다.
- 패키지 원칙이 저장소나 문서 형식의 전용 계약을 덮어쓰지 않는지 명시 호출 평가로 검증한다.
