# Tighten Docs

[한국어](README.md) | [English](README.en.md)

## 요약

`tighten-docs`는 이미 선택되었거나 구체적으로 요청된 문서, diff, 문서 패키지를 분명한 현재 최종 상태로 만드는 스킬입니다. 문장 수준에서는 표현을 다듬고, 패키지 수준에서는 역할, 독자 경로, 현재 소유권을 정리합니다.

핵심은 다음 기준입니다.

- 최신 지적을 최종 문장으로 반영한다.
- 작성 제약을 처음부터 본문에 적용한다.
- 삭제 요청은 실제 삭제로 반영한다.
- 현재 정본을 증거, 로그, 초안, 오래된 자료와 구분한다.
- 같은 현재 결정, 계약, 게이트, 계획의 소유 문서를 하나로 만든다.
- 레거시 호환 요구는 확인된 경우만 남긴다.
- 문서의 요구 언어와 로캘을 지킨다.
- 원문에 없는 사실, 정책, 결정, 구현 의미는 만들지 않는다.

## 빠른 시작

**설치**

```bash
npx skills add perhapsspy/tighten-docs
```

혹은 `skills/tighten-docs` 폴더를 에이전트 스킬 디렉터리에 직접 복사합니다.

**바로 사용**

```text
$tighten-docs 이 문서를 방금 지적한 내용대로 더 짧게 고쳐줘.

$tighten-docs 이 diff에서 삭제 요청을 실제 삭제로 반영해줘.

$tighten-docs 이 문서 초안을 요구된 본문 언어와 짧은 최종 문장으로 작성해줘.

$tighten-docs 이 README, 설계 문서, 런북 패키지의 중복 소유 문서와 오래된 경로를 정리해줘.

$tighten-docs 이 마이그레이션 문서 패키지를 읽기 전용으로 리뷰하고 이슈와 리스크부터 알려줘.
```

## 이런 때 사용

- 작성하거나 고칠 문서나 diff가 이미 구체적으로 정해져 있을 때
- 지적을 본문에 바로 반영해야 할 때
- 초안 작성 시점부터 직접적인 문장, 요구 언어, 중복 라우팅 정리를 적용해야 할 때
- 삭제 요청이 경고, 라우팅, 방어적 문구로 되살아날 때
- 문서 언어는 지키고 코드 식별자, 경로, 제품명은 그대로 둬야 할 때
- 선택된 문서 패키지에서 진입점, 독자 경로, 현재 소유 문서가 충돌할 때
- 증거, 로그, 초안, 보관 자료가 현재 정본처럼 보일 때
- 런북의 사전 조건, 중단 조건, 승인 경계, 증거, 롤백, 복구를 점검할 때
