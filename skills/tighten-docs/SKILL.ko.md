# Tighten Docs

## 목적

선택된 문서, 문서 diff, 문서 패키지를 하나의 분명한 현재 최종 상태로 만든다. 문장 수준에서는 표현을 다듬고, 패키지 수준에서는 역할, 독자 경로, 현재 소유권을 명확히 한다.

원문의 의미를 보존한다. 사실, 정책, 약속, 담당, 날짜, 제약, 결정, 구현 의미를 새로 만들지 않는다.

## 우선순위

서로 겹치면 다음 순서로 적용한다.

1. 사용자와 저장소 지침
2. 선택된 문서 형식의 전용 계약
3. 확인된 기준 근거
4. 아래의 일반 문서 규칙

상위 근거가 필요한 선택을 정하지 않았다면 조용히 결정하지 말고 짧게 확인하거나 결정 요청으로 남긴다.

## 모드

- **대상 편집**: 구체적인 문서나 diff 하나를 작성하거나 수정한다.
- **대상 리뷰**: 구체적인 문서나 diff 하나를 변경 없이 리뷰한다.
- **패키지 정리**: 선택된 문서 집합을 편집, 이동, 격하, 삭제하거나 연결한다.
- **패키지 리뷰**: 선택된 패키지의 소유권, 탐색 경로, 오래된 상태, 중복, 운영 완결성을 변경 없이 리뷰한다.

리뷰는 읽기 전용으로 본다. 편집이나 정리는 선택된 문서 범위만 바꿀 권한으로 본다. 문서 생성, 이동, 삭제는 사용자가 변경을 요청했고 적용되는 계약이나 근거가 그 동작을 정당화할 때만 수행한다.

## 실행 흐름

1. 선택된 범위, 모드, 최신 지적이나 작성 제약, 요구 본문 언어와 로캘을 고정한다.
2. 일반 패키지 원칙보다 저장소와 문서 형식의 전용 계약을 먼저 읽는다.
3. 각 대상의 단일 역할, 핵심 주체, 독자 행동, 문서가 책임지는 약속, 함께 바뀌는 일관된 단위를 확인한다.
4. 패키지라면 진입점, 라우터, 현재 소유 문서, 게이트/런북, 백로그, 증거/로그, 작업/보관 문서 사이의 실질적인 충돌만 그린다.
5. 관련 단위를 유지, 교체, 삭제, 압축, 언어 수정, 이동, 격하 중 하나로 처리하고 최종 상태 문장을 바로 쓴다.
6. 미확정 상태와 원문 의미를 보존한다. 정해지지 않은 소유권이나 정책 선택은 결정 요청으로 남긴다.
7. 현재 사실, 현재 약속별 단일 소유 문서, 분명한 독자 경로, 정당한 문서 경계, 요구 언어, 유효한 링크, 리뷰 가능한 밀도를 다시 확인한다.

## 문장 규칙

- 새로 요청된 대상을 작성할 때도 최신 지적이나 작성 제약을 처음부터 최종 문장으로 반영한다.
- 거절되었거나 오래됐거나 제외된 내용은 삭제한다. 경고나 사유는 대상이 그 목적을 맡을 때만 남긴다.
- 레거시 호환, 대체 경로, 예외, 라우팅 주장은 선택된 근거가 현재 필요성을 증명할 때만 남긴다.
- 국소 교정은 사용자가 더 넓은 정책을 명시적으로 요청하지 않는 한 국소적으로 유지한다.
- 방어적 설명, 중복 경로, 작업 과정 설명, agent 편의 메모, 사과형 문장, 편집 메타 설명은 대상이 그 목적을 맡지 않으면 삭제한다.
- 공개 진입점이 독자의 평가·선택·사용 시작을 돕는다면 확인된 사용자 상황·결과·다음 행동을 먼저 제시한다. 내부 절차나 메커니즘은 그 판단에 도움이 될 때만 남긴다.
- 선택된 대상, 저장소나 사용자 지시, 최신 지적, 작성 제약에서 요구 언어와 로캘을 판단한다. 서로 충돌하면 짧게 확인한다.
- 유지할 일반 문장은 요구 언어로 고치고, 다른 언어의 중복되거나 오래됐거나 여분인 일반 문장은 삭제한다.
- 코드 식별자, API 이름, 파일 경로, 명령어, 제품명, 도메인 용어, 인용된 프로토콜 문구, 외부 소유 라벨은 그대로 둔다.

## 패키지 규칙

다음 역할은 고정 템플릿이 아니라 판단 기준으로 쓴다.

- **진입점**: 패키지의 목적, 현재 상태, 시작점을 말한다.
- **라우터**: 독자 작업을 문서에 연결하고 각 파일의 책임을 말한다.
- **현재 소유 문서**: 현재 주제, 계약, 결정, 게이트, 계획 하나를 소유한다.
- **게이트/런북**: 사전 조건, 허용 동작, 중단 조건, 승인 경계, 증거, 롤백, 복구를 소유한다.
- **백로그**: 실시간 체크리스트, 삭제형 할 일, 역사적 감사 중 성격을 밝힌다.
- **증거/로그**: 증거나 시간 순서를 보존하되 현재 정본처럼 보이지 않게 한다.
- **작업/보관 문서**: 미확정 또는 오래된 자료를 현재 문서와 분리한다.

현재 정본은 증거, 시간 순서, 초안, 보관 자료와 분리한다. 현재 결정, 계약, 게이트, 계획 하나에는 소유 문서 하나만 둔다.

문서의 부분이 독립적으로 유용한 독자 행동, 소유권, 변경 이유를 가질 때만 분리한다. 독자가 함께 사용하거나 하나의 현재 계약·절차를 완성하는 내용은 함께 둔다. 길이, 제목 개수, 개념 이름만으로 분리하지 않는다.

실제 대상이 바뀔 때 참조와 동시 수정은 허용한다. 경쟁하는 정본 정의, 결정, 계약, 절차는 피하되, 일반적인 문서 간 링크나 필요한 동시 수정을 그 자체로 결함으로 보지 않는다.

독자가 재개, 결정, 구현, 검증, 운영 중 무엇을 하려는지에서 시작한다. 선택된 근거와 계약이 정당화하는 경로와 문서 표면만 만든다.

소유권이 이동하면 이전 문서는 새 소유 문서로 연결하거나 역사/보관 상태를 밝혀 정본처럼 경쟁하지 않게 한다. 라우터는 독자, 선택 기준, 순서, 전제조건을 소유할 수 있지만 경쟁하는 정본 소유자가 되지 않는다. 다른 곳에 설명을 더하기보다 기존 소유 문서를 먼저 고친다.

이메일, 체크리스트, ADR, 런북, 인수인계, 로그처럼 유용한 고유 형식은 보존한다. 역사 정리를 명시적으로 요청하지 않았다면 보기 좋게 만들려고 append-only 로그를 다시 쓰지 않는다.

재구성이 독자 행동, 소유권, 운영 사용성을 실질적으로 개선하지 않으면 변경하지 않거나 가볍게만 고친다.

## 출력 계약

- 편집이나 정리 작업은 가능하면 선택된 파일을 직접 바꾸고, 달라진 소유 문서나 경로와 막힌 결정만 보고한다.
- 리뷰는 사전 요약 없이 심각도 순의 이슈와 리스크로 응답을 시작하고, 근거를 필요한 변경이나 결정 요청에 연결한다.
- 응답은 작업 크기에 맞춘다. 요청받지 않은 별도 리뷰 보고서, 작업 문서, 문서 표면을 만들지 않는다.

## 경계

다음이 필요하면 멈추고 확인하거나 넘긴다.

- 외부 기준 소스 선택
- 제품, 정책, 아키텍처, 구현 결정 발명
- 작업 상태, 작업 로그, 인수인계 저장 구조 설계
- 구현 계획의 의미나 범위 변경
- 순수 번역이나 지역화
- 항상 읽히는 저장소 지침 파일의 의미나 구조 결정

이미 선택된 지침이나 계획의 구체적인 문장만 다듬는 것은 가능하지만, 그 문서가 무엇을 지배하는지 바꾸지는 않는다.
